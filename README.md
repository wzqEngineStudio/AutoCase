# AutoCase

**面向个人开发者的 AI 用例驱动自动化测试管理系统**

> 传统用例/缺陷管理平台因项目保密限制无法接入 AI，用例全靠人工逐字段填写。AutoCase 的核心理念是：**只要 AI 生成的测试用例符合规范格式，系统就能自动识别、解析并执行**——让 AI 成为你的测试用例生成器，让系统成为执行器。

---

## ✨ 特色定位

| | 传统用例管理平台 | AutoCase |
|---|---|---|
| **目标用户** | 团队/企业 | **个人开发者** |
| **用例来源** | 人工逐条填写 | **AI 生成 + 规范识别** |
| **数据同步** | 与研发/项目管理平台打通 | **本地化管理，无外部依赖** |
| **提单流程** | 缺陷→指派→修复→验证闭环 | **无提单功能（不对接开发）** |
| **部署方式** | SaaS / 私有化服务器 | **桌面端应用，零配置启动** |

### 🎯 核心能力

- **AI 用例原生兼容** — 系统以 `*Case.json` 文件为用例载体，只要 AI 输出的 JSON 符合规范即可直接导入使用，无需人工二次录入
- **多语言脚本执行引擎** — 内置 Python / GDScript / Java / C / C++ / Go / Rust / JS / TS / Ruby / Shell 等语言的执行模板，支持并行执行、超时熔断、暂停恢复
- **用例-脚本自动关联** — 通过名称模糊匹配自动将测试脚本与用例 JSON 绑定，执行结果自动回写用例状态
- **手工+自动化双轨** — 支持传统手工测试任务管理和全自动脚本批量执行两套工作流
- **完全离线运行** — 数据存储在本地 H2 数据库和 JSON 文件中，不依赖任何远程服务

## 📁 功能模块

### 1. 用例管理
- 以文件目录树浏览和管理 `*Case.json` 用例文件
- 支持新建、编辑、删除、筛选用例
- 用例属性：用例名、优先级(P0-P3)、严重程度、版本、分组、前置条件、测试步骤、预期结果、环境约束等
- 支持用例间**关联关系**和**互斥关系**定义

### 2. 自动化测试
- 自动扫描目录下的测试脚本文件
- 可视化选择脚本后一键批量执行
- 多线程并行执行，支持优先级排序（P0 优先 → 步骤数短优先）
- 执行超时熔断（默认 60s）、崩溃检测、暂停/停止控制
- 通过 `PASS` / `FAIL` / `BLOCK` 关键字或退出码判定结果
- Debug 日志自动收集与过期清理
- 执行完成后自动回写关联用例状态

### 3. 手工测试
- 树形任务组 → 任务 → 用例执行记录的三层结构
- 从用例管理面板快速添加用例到测试任务
- 逐条执行并记录实际结果、失败原因、截图附件
- 支持 PASS / FAIL / BLOCKED / SKIPPED 等状态标记

### 4. 缺陷管理
- 本地缺陷库，记录用例执行中发现的问题
- 支持严重程度、优先级、状态跟踪
- ⚠️ **不含提单功能**——本系统面向个人开发者，不与任何 Bug 跟踪或项目管理系统对接

### 5. 测试报告
- 自动汇总执行结果，生成通过率统计
- 基线快照记录（OS / Java 版本 / Godot 版本 / MD5 等）
- 按严重程度和优先级维度展示分布

### 6. Git 集成
- 内置 GitHub 操作面板，支持一键 Push 到远程仓库
- 用例以 JSON 文件形式纳入版本管理

### 7. 设置与持久化
- 自定义各语言执行命令模板
- 自定义通过/失败/阻塞判定规则
- UI 布局位置、最近打开目录等状态自动保存

## 🚀 快速开始

### 环境要求

- **JDK 17 或更高版本**
- Maven 3.6+（如从源码构建）

### 构建运行

```bash
# 克隆仓库
git clone https://github.com/wzqEngineStudio/AutoCase.git
cd AutoCase

# 打包（含所有依赖的可执行 JAR）
mvn clean package

# 运行
java -jar target/autocase-1.0.0.jar
```

### 使用流程

1. **准备用例** — 手动编写或让 AI 生成符合规范的 `*Case.json` 文件，放入项目目录
2. **选择根目录** — 在 AutoCase 中打开包含用例的根目录，系统自动扫描识别
3. **编写测试脚本**（可选）— 如需自动化执行，在同级目录放入对应语言的测试脚本
4. **执行测试** — 选择「自动化测试」批量跑脚本，或选择「手工测试」逐条执行
5. **查看报告** — 在「测试报告」面板查看执行统计和详情

## 📖 用例 JSON 规范

AutoCase 以 `*Case.json` 文件作为用例的标准载体。一个最小可用示例：

```json
{
  "casesID": "TC001",
  "group": "用户模块",
  "version": "1.0",
  "severity": "High",
  "priority": "P0",
  "caseInfo": {
    "caseName": "登录成功",
    "precondition": "用户已注册",
    "steps": ["输入正确账号密码", "点击登录按钮"],
    "testData": { "username": "admin", "password": "123456" }
  },
  "expectedResult": {
    "expectedOutput": "跳转到首页",
    "acceptanceCriteria": "页面显示用户昵称"
  }
}
```

> 只要 JSON 文件名以 `Case.json` 结尾且包含上述核心字段，AutoCase 就能自动识别。额外字段会保留为动态测试数据，不影响解析。

## ⚠️ 设计边界

本系统围绕**个人开发者**场景设计，以下能力**不在范围内**：

- ❌ 不提供缺陷提单 / 工单流转（不对接 Jira、禅道、Tapd 等）
- ❌ 不提供多人协作 / 权限管理
- ❌ 不与研发代码仓库做 CI/CD 集成
- ❌ 不做服务端部署（纯桌面端应用）

如果你需要团队协作式的测试管理平台，建议使用 [TestLink](https://github.com/testlinkorg/testlink-code)、[Zentao](https://www.zentao.net/) 等方案。
